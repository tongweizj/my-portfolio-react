1. 统一 API 调用 (api/)
   动作：创建一个 api/axiosInstance.js。

目的：配置 baseURL: '/api' 并设置 withCredentials: true。

关键点：在这里写拦截器。当后端返回你定义的 AppError（如 401）时，前端能在这里统一捕捉，而不是在每个组件里写 try-catch。

## 1. 基础设施层：API 客户端 (Axios)

不要在组件里直接写 `axios.get('...')`。

- **逻辑**：

创建一个统一的 `api/` 目录，配置 **Axios 拦截器 (Interceptors)**。

- **作用**：

统一处理你刚才在后端定义的 `AppError`。
比如捕获到 `401` 就自动清理 Cookie 并跳转登录，捕获到 `500` 就弹出错误提示。

---

在现代 React 重构中，**统一 API 管理** 是建立一套**从请求发送到错误响应的完整链路**。

这套体系能让你在前端精准对接后端重构后的 `AppError`，并根据状态码（如 401, 403）自动执行逻辑（如跳转登录）。

---

## 1. 核心逻辑：分层架构

我们将 API 逻辑分为三个层级，以保证代码的**低耦合**和**高复用**：

1. **Axios 实例层**
全局配置（Base URL、超时、凭证）。

2. **拦截器层 (Interceptors)**：
统一处理请求头（Token）和响应错误（AppError）。

3. **服务模块层 (Services)**：
按业务功能（User, Article）编写具体的请求函数。

## 2. 第一步：创建 Axios 实例与拦截器

在 `src/api/` 目录下创建 `axiosInstance.js`。

```javascript
import axios from 'axios';

const api = axios.create({
  baseURL: '/api', // 对应你后端的路由聚合前缀
  timeout: 5000,
  withCredentials: true, // 必须开启，否则无法发送和接收 Cookie 里的 Token
});

// 请求拦截器：发送请求前自动处理
api.interceptors.request.use(
  (config) => {
    // 可以在这里统一添加 Header，或者处理加载状态
    return config;
  },
  (error) => Promise.reject(error)
);

// 响应拦截器：统一接住后端的 AppError
api.interceptors.response.use(
  (response) => {
    // 如果后端返回的是标准的 { status: 'success', data: ... }
    return response.data;
  },
  (error) => {
    // 这里处理所有非 2xx 的状态码
    const { response } = error;

    if (response) {
      // 1. 自动处理 401 (未登录/Token过期)
      if (response.status === 401) {
        console.warn('登录已过期，正在跳转...');
        window.location.href = '/login';
      }

      // 2. 自动提取后端 AppError 里的 message
      const errorMessage = response.data?.message || '服务器发生错误';

      // 你可以在这里配合 UI 组件（如 Toast/Message）弹出报错
      // alert(errorMessage);

      return Promise.reject(response.data);
    }

    return Promise.reject(error);
  }
);

export default api;
```

### 3. 第二步：按功能模块封装请求

不要在组件里写路径。在 `src/features/` 下对应的模块中创建 API 文件。

#### 示例：`src/features/articles/articleApi.js`

```javascript
import api from '../../api/axiosInstance.js';

export const getArticles = () => api.get('/articles');

export const getArticleById = (id) => api.get(`/articles/${id}`);

export const createArticle = (data) => api.post('/articles', data);

export const updateArticle = (id, data) => api.put(`/articles/${id}`, data);
```

---

### 4. 第三步：在组件中使用

重构后，你的业务组件变得极其清爽，只需关注数据和状态。

```javascript
import { useEffect, useState } from 'react';
import { getArticles } from '../features/articles/articleApi';

const ArticleList = () => {
  const [articles, setArticles] = useState([]);
  const [error, setError] = useState(null);

  useEffect(() => {
    const fetchArticles = async () => {
      try {
        const data = await getArticles();
        setArticles(data); // 拦截器已经处理了 .data，这里直接拿结果
      } catch (err) {
        // 这里的 err 就是拦截器 reject 出来的 AppError 对象
        setError(err.message);
      }
    };
    fetchArticles();
  }, []);

  if (error) return <div>报错了: {error}</div>;
  // ...渲染逻辑
};
```

---

### 💡 为什么这种重构逻辑对你很重要？

1. **路径解耦**：如果你决定把接口从 `/api/articles` 改为 `/api/v2/articles`，你只需要改 `articleApi.js` 这一处。
2. **错误处理自动化**：由于你之前在后端花费精力编写了 `AppError`，通过 Axios 拦截器，前端可以**一次性**处理所有的 401、403、500 错误，而不需要在每个 `useEffect` 里写重复的 `if (status === 401)`。
3. **配合目录重构**：这种结构完美对应你刚才在后端完成的“聚合路由” (`routes/index.js`) 逻辑，前后端架构会变得高度对称。
